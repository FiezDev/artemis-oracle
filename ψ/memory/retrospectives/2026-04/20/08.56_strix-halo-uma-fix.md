# Session Retrospective — Strix Halo UMA Memory Freeze Diagnosis & Fix

## Session Summary
Diagnosed recurring PC freezes on Strix Halo (Ryzen AI Max+ 395, 128GB UMA) that were causing hard reboots every ~1 hour. Traced root cause to BIOS VRAM split of 96/32 starving Windows of RAM. Applied 64/64 split and updated `.wslconfig` from 22GB to 32GB WSL cap.

## What Got Done
- Checked dmesg, journalctl, and Windows Event Viewer for OOM/crash evidence
- Found two Kernel Event 41 (Critical) hard reboots in last 3 hours — no BSOD, no BugCheck, consistent with UMA memory exhaustion
- Mapped full memory layout: 96GB GPU VRAM → only 32GB visible to Windows → WSL takes 22GB → 10GB for Windows → freeze
- Audited all WSL processes: 3 VS Code workspaces (7GB), 3 Claude sessions (1.6GB), Next.js (1.8GB), chrome-devtools-mcp×4 (1.4GB), agent-browser (0.9GB)
- Checked Windows pagefile: 100GB configured, barely used (0.62GB) — confirms freeze happens too fast for pagefile to help
- Applied BIOS VRAM change from 96/32 to 64/64
- Updated `.wslconfig` from `memory=22GB` to `memory=32GB`
- Verified new state: Windows 63.6GB (34GB free), WSL 31GB (14GB free)

## Lessons Learned
- Strix Halo UMA architecture: BIOS VRAM reservation is a *minimum* that steals from system RAM. 96GB fixed reservation left only 32GB for entire OS
- UMA freeze signature: Kernel Event 41 with no BugCheck/Event 1001 = hard hang, not BSOD. System can't write events before dying
- Windows pagefile is useless if the freeze is instantaneous — need physical RAM headroom
- VS Code extension host is the single biggest WSL memory consumer (3.2GB for one workspace with 12+ extensions)
- Multiple Claude sessions × chrome-devtools-mcp adds up fast: 3 sessions = ~3.2GB of MCP processes
