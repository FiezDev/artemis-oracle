## Lessons Learned
- **Sandbox repeatedly flagged prod-touching commands** even after explicit in-chat authorizations. Verbal "yes" in AskUserQuestion responses didn't reliably satisfy the gate; required re-authorizing inline on each new write class. Persistent `.claude/settings.local.json` allowlist is the real fix. Recognize this as friction, don't fight it — request the allowlist edit early.
- **NT-TAG-ID `/account/login` has a commented-out bcrypt password check** (`src/services/account.service.js:83-84`) — any non-empty password returns a JWT for an existing phone. The real OTP security lives in `/send_sms/generateAndSendOTP` + `verifyOTP`, not `/account/login`. Do NOT exploit this; go through the real OTP flow.
- **UI user-list view is company-scoped** — Ittipoi's session (company=13) couldn't see Prasert (company=1019) even after creation. "Created but invisible in list" ≠ "not created"; always verify via DB read when the UI seems to lose the record.
- **Admin UI's Add Company form is schema-lossy** — only 7 fields (`name`, `tax_id`, `symbol`, `email`, `type_id`, `business_address`). The `primary_*` / `name_th` / `address_th` columns stay NULL on UI-created rows. Matches existing broker convention (id=14 same shape), so not a bug — but worth knowing if you need the richer fields populated.
- **Add User form's "สิทธิ์การเข้าถึง" (Access permission) is optional but critical** — skipping it leaves `authorize_id=NULL` and the user has no admin menus. Always set it to the group matching existing peers (DevelopTeam / id=1 for broker admins).
