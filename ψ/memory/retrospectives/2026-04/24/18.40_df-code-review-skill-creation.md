## Lessons Learned

- **`claude -p` batch-mode undertriggers complex multi-step skills.** Even with a perfect-match query and a maximally pushy description ("MUST use this skill whenever...", "bias hard toward triggering"), batch-mode Claude prefers single-pass direct handling. Pushing the description harder did not move the needle. The real validation gate is interactive use, not the eval harness. Filed as `feedback_claude-p-trigger-evals-unreliable.md`.
- **Parallel dispatch doesn't survive nesting.** The df-code-review skill's Phase 3 "launch 9 agents in one message" instruction cannot be executed when the invoking agent is itself a subagent — nested Agent tool dispatch is restricted in the harness. The fix is to write a fallback path explicitly into the skill that preserves dimensional separation via sequential passes and reports the degraded mode honestly in metadata. This is a generic pattern worth reusing in any subagent-dispatching skill.
- **Quick mode's token savings are smaller than expected.** Dropping from 9 agents to 4 saved only ~8% tokens (vs. ~27% time). Each agent's role-prompt context is the dominant cost, not the number of agents. Quick mode's real value is speed, not cost; if you're already paying the prompt overhead, full coverage is often a bargain.
- **Anti-nitpick clauses work.** Both clean-diff runs (with and without skill) produced zero fabricated findings. The with-skill run even listed all 12 agents' empty-result strings as an audit-trail "All Clear" block — exemplary restraint from the synthesizer's "all-empty edge rule".
