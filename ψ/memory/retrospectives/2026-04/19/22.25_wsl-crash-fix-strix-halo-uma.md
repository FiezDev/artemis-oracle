# WSL Crash Fix ‚Äî Strix Halo UMA Memory Misconfiguration

## Session Summary

User reported the system "clashed recently." Investigation confirmed **WSL crashed at 17:56 today (2026-04-19)** on the Beelink GTR9 Pro workstation, followed by four back-to-back reboots (18:06, 18:08, 21:27, 21:52). Root cause traced to a `.wslconfig` misconfiguration interacting badly with the Strix Halo unified memory architecture. Wrote a corrected `.wslconfig`, verified it took effect, and flagged a separate performance concern about ComfyUI running on DirectML instead of native ROCm.

## What Got Done

### Diagnosis
- Kernel log (`dmesg`) showed DNS/networking flapping and WSL warnings around the crash window
- `last` command confirmed unclean tmux termination at the crash timestamp
- `ps -p 4252` confirmed stale `scheduled_tasks.lock` referenced a dead process
- PowerShell queries exposed the hardware/config mismatch:
  - **128 GB DDR5 physically installed** (8 √ó 16 GB Micron @ 8000 MT/s)
  - **Windows 11 Pro only sees 32 GB** (UMA BIOS split: 96 GB ‚Üí VRAM, 32 GB ‚Üí system)
  - **Only 1.8 GB free at Windows level** ‚Äî vmmem starving
  - Motherboard `MaxCapacity` reports 64 GB (BIOS-level quirk, not our problem)

### Root Cause
Old `.wslconfig` contained:
- `memory=96GB` ‚Äî requesting 3√ó more than Windows could provide
- `[boot]`, `[user]`, `[network]` sections ‚Äî invalid in `.wslconfig` (belong in `/etc/wsl.conf`)
- `networkingMode=mirrored` misplaced under `[network]` (should be `[wsl2]`)
- `swap=64GB` ‚Äî oversized, bloats ext4.vhdx

When WSL tried to allocate 96 GB from a 32 GB Windows pool, vmmem got OOM-killed ‚Üí WSL crash ‚Üí cascade of failed reboots.

### Fix Applied
Wrote new `C:\Users\bjgdr\.wslconfig`:
```ini
[wsl2]
memory=22GB
processors=16
swap=32GB
networkingMode=mirrored
autoMemoryReclaim=gradual
sparseVhd=true
pageReporting=true

[experimental]
hostAddressLoopback=true
```

Backed up old config as `.wslconfig.bak-2026-04-19`. User ran `wsl --shutdown`; verified post-restart:
- `free -h`: 21 GB total (was 30 GB), 32 GB swap
- `nproc`: 16 (was 32)
- Windows now has ~10 GB headroom (was 1.8 GB)

Also cleared stale `.claude/scheduled_tasks.lock` (pid 4252 dead).

## Lessons Learned

### 1. Unified Memory Architecture constraints are invisible from WSL
WSL `free -h` reported 30 GB as if the host had 30 GB of RAM. It doesn't know about the 96 GB sitting on the GPU side of the UMA split. Any `.wslconfig` memory sizing **must** match what Windows sees (via `Get-CimInstance Win32_OperatingSystem`), not what DIMMs physically show.

### 2. `.wslconfig` vs `/etc/wsl.conf` are different files with different sections
- `.wslconfig` (Windows side): `[wsl2]`, `[experimental]` only
- `/etc/wsl.conf` (Linux side): `[boot]`, `[user]`, `[network]`, `[automount]`, `[interop]`
- Mixing them means settings are silently ignored. Previous config had `[boot] systemd=true` in `.wslconfig` ‚Äî a no-op. Fortunately also present in `/etc/wsl.conf`, so systemd actually worked.

### 3. `autoMemoryReclaim=gradual` + `pageReporting=true` prevent vmmem hoarding
Without these, WSL holds onto RAM it isn't using, starving Windows and triggering OOM crashes. Essential on UMA systems where the system-RAM budget is small.

### 4. Container memory stats can mislead
`docker stats` showed the ComfyUI container using 756 MB / 30.99 GiB ‚Äî the 30.99 GiB limit was just WSL's total, not a real container cap. `HostConfig.Memory=0` confirms no actual limit. On crash-prone systems, adding a container-level `mem_limit` is cheap insurance.

### 5. DirectML vs Native ROCm = major perf gap, currently unexploited
The ComfyUI container (`comfyui-gfx1151-wsl2:latest`) uses `/dev/dxg` (Windows graphics passthrough). Native ROCm (`/dev/kfd` + `/dev/dri`) is not available in the current setup despite wiki notes claiming 110 GB ROCm support. Per user's own concept `native-rocm-vs-directml`, native ROCm "massively outperforms DirectML." Separate investigation needed.
