# Session Retrospective — claude-code-setup v1.0.0 (zero → ship)

## Session Summary

Built `claude-code-setup` from an empty repo to a tagged v1.0.0 release in one session. Modular, idempotent installer that ports a personal Claude Code dev environment (config, 268 skills, 16 plugins, 4 CLIs, shell snippet, 2 Docker DB stacks with snapshot/restore) to a fresh Linux/WSL/macOS machine. Six batches, ~62k LOC across 346 files, 92 bats integration tests passing, all on `~/dev-personal/claude-code-setup/`.

## What Got Done

- **Brainstorm + spec** — section-by-section design via brainstorming skill: 5 scoping Q&As, 2 approach options, 13-section spec doc reviewed and refined, committed at `docs/superpowers/specs/2026-05-06-claude-code-setup-design.md`.
- **6 batches, 6 plans, 6 tags** — `batch-1` (lib + bootstrap), `batch-2` (claude-config/skills/shell), `batch-3` (plugins, --json driven), `batch-4` (cli-tools, brew + symlink), `batch-5` (DB modules + snapshot/restore), `batch-6` (strp design doc + e2e smoke + README + v1.0.0). Each batch followed write-plan → reviewer → fix → subagent-driven-execute → cross-cutting reviewer → backport.
- **Real bugs caught and fixed mid-execution, all backported to plans:**
  - Batch 2: `((j++))` returning 0 on first iteration aborted under `set -e`; `local` keyword inside pipe subshell errored
  - Batch 3: `@tsv` with empty pin field collapsed under bash IFS — switched to `\x01` SOH delimiter
  - Batch 4: brew Cellar symlink wasn't idempotent (different paths resolving to same canonical file); installer counter semantics conflated install with skip
  - Batch 5: `verify_manifest` `cmd | while` subshell ate `fail=1` — fixed with `< <(...)` process substitution; `RESTORE_FROM` set in bootstrap.sh but not exported, silently no-oping the entire restore path
  - Batch 6: README example used short module IDs (`10,30,50`) but bootstrap requires full directory names (`10-claude-config,...`) — first command a new user would copy was broken
- **Subagent-driven execution** — 7 implementer + 7 spec/quality reviewer + 6 cross-cutting reviewer dispatches. Pattern: provide full task text + context per dispatch (no "go read the plan"), file-based fixtures over env-var propagation, haiku for mechanical tasks, sonnet for logic-heavy ones.
- **End-to-end live smoke verified** twice on isolated `$HOME` — clean install of network-free subset (10/20/50/90), 268 skills, idempotent re-run, doctor lists all modules.

## Lessons Learned

- **Cross-cutting reviewers find class of integration bugs that per-task reviewers miss.** Every batch's final `feature-dev:code-reviewer` pass over `batch-N-1..batch-N` caught at least one real load-bearing bug (counter semantics, RESTORE_FROM export, etc.). The per-task reviews focused on whether the implementer met the prompt; the cross-cutting pass asks whether the modules wire together. Memory `feedback_final-cross-cutting-review` reinforced.
- **Reviewers can be confidently wrong about invisible bytes.** Twice (batch-3, batch-6) a reviewer flagged `join("")` as "empty delimiter — critical bug" when the source file actually contains `join("` + 0x01 + `")` (SOH byte renders as nothing). Always verify with `xxd` and check whether the test that *would* fail under the claimed bug actually passes. Don't blindly apply reviewer fixes.
- **File-based test fixtures beat env-var propagation through `bash -c`.** Batch-3 reviewer caught that `MOCK_PLUGIN_LIST` env vars don't reliably reach a stub script run inside `bash -c "..."` indirection. Switched all subsequent batches to write fixtures into `$BATS_TEST_TMPDIR/<name>.txt` (or `.json`) and have the stub `cat` them. Robust + lets tests modify mid-test.
- **`jq -c '.tools[]'` per-row JSON > IFS+delimiter tricks.** Batch-3 used `\x01` SOH byte (works but invisible/fragile in editors). Batch-4 reviewer pushed for `jq -c` per-row pattern: emit one compact JSON per line, query fields with `jq -r '.field' <<<"$row"`. Costs ~16 extra jq invocations per run (negligible). Source survives any text editor.
- **Default to dry-run for destructive tools.** `restore.sh` requires `--confirm-destructive` (or `BOOTSTRAP_CONFIRM_DESTRUCTIVE=1` env). Default mode prints what it *would* do — checksums verified before any container is touched. Cost: one extra flag from automation. Benefit: no accidental volume wipes.
- **Backport mid-execution fixes to source plan immediately.** Memory `feedback_backport-fixes-to-source` already captures this. Reinforced when we found the batch-3 reviewer's "fix" lived only in the implementation, never in the plan — re-running from the plan would reproduce the bug. Now: every fix → patch the plan in the same commit.
- **`--restore-from` was a critical integration bug because tests didn't cover the bootstrap → install.sh chain.** Per-task tests called `restore.sh` directly with env vars set in the test harness. The cross-cutting reviewer noticed nobody tested the actual user-facing path. Added a regression test that creates a temp module asserting RESTORE_FROM + BOOTSTRAP_CONFIRM_DESTRUCTIVE both reach the install subshell.
