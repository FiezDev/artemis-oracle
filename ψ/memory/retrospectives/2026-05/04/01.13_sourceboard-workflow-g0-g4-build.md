## Lessons Learned

- **Two-postgres realities.** The host DB at 5432 (where `bun run db:migrate` lands by default via `.env`) is separate from the docker-compose DB at 5532 (which the running dashboard reads). Migrations need to run against both with `DATABASE_URL=postgresql://qone:qone123@localhost:5532/...`. Also: 450 polluted workflows on the host DB pre-date the test-cleanup fix; 4 clean ones on docker. Tests aren't affected (they use `test-WF-*` prefixes) but it's a divergent state to be aware of.

- **Test fixtures must use the seed-helper pattern from day one.** The 207-workflow leak was caused by tests calling the service directly with auto-generated UUIDs that didn't match `id LIKE 'test-%'`. The seedWorkflow/seedProcess helpers force the prefix and are mandatory for any new test.

- **Dev-mode flakiness defeats Playwright snapshot determinism.** `next dev` compiles routes lazily; concurrent first-hit requests + HMR cause the click→render timing to drift between runs. Production build (`next build && next start`) makes snapshots deterministic with workers=1. Cost: a longer baseline-update cycle, but worth it for stable visual regression.

- **Conditional dispatch is cleaner as engine-internal.** Initial design tried having conditionals create step_runs that "auto-complete." Settled on no-step-run, no-task at all — `advanceAfterNode` evaluates inline against `workflow_run.variables`, then walks both direct and indirect downstream (those whose dep chain routes through a conditional whose own dep includes currentNode). The candidate-set-widening was the key insight that made non-blocked branches transition cleanly.

- **The Critical-#3 class (`goals` table never created by hand-written migration) is a structural recurrence risk, not a one-off.** Patched `task_comments` in batch 3 (R-5 from re-review). Wrote a CI drift-checker so the next instance fails loudly. Same root cause: `migrate.ts` ignores Drizzle's `_journal.json` and snapshot.json. Lessons compounded across batches.

- **Subagent fan-out hits usage limits unpredictably.** First /df-code-review fan-out (13 agents) returned everything cleanly; later batches saw subagents 504-out and forced fallback to inline edits. The pattern: when subagents fail, don't keep retrying — just do the work directly and keep moving.

- **Auto-continue mode produces real throughput when discipline holds.** User's choice of "auto-continue G0→G4, only stop on red" let the build run continuously. The discipline that made it work: drift check + tests + tsc after every batch; commit + merge with --no-ff between batches; surface scope-reduction calls explicitly (G3 deferred AI Search auto-merge; G4 deferred new scenario tests + 2 snapshots). Don't pretend everything got done.
