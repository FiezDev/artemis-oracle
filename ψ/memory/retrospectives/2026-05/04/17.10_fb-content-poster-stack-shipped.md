## Lessons Learned

- **Final cross-cutting reviews catch what per-task reviews miss.** Both G5 and G6 had per-task reviews approving cleanly, then a final whole-branch review found 3 Criticals each (state-machine bugs, type drift, race conditions). The cost of one extra review is small; the cost of those bugs in production is large. This is now a memory.
- **Live-running the pipeline surfaces bugs unit tests can't.** B1, B2, B3 all passed every per-task and cross-cutting review. They surfaced only when running a real YouTube URL through the engine end-to-end. Integration tests with real DBs and worker callbacks are the right level of confidence.
- **Direct-to-main workflow is faster but demands stronger discipline.** When the user said "no more PRs," the velocity went up but every commit became its own gate. Tests-must-be-green-before-commit was non-negotiable. Subagents responded to this by running full suites before each commit.
- **Operational discovery is half the work.** SERVICE_SECRET wired in 3 places, two postgres instances at different ports, dashboard.bun.lock vs api/bun.lock, hardcoded `/home/bjgdr/.linuxbrew` paths, agent runners running but not consuming because of missing env. None of these surfaced until live testing.
- **The "stale docker container" pattern repeats.** Three times today: G7 first run saw `[null]` output (image was pre-G7), bug 1 needed frontend rebuild to verify, B3 needed api rebuild. Always verify what the running container has, not what's on disk.
- **Plan-per-batch over plan-everything-up-front works.** Each batch's plan benefited from real lessons from the previous batch — API signatures, naming conventions, hidden dependencies. G5's plan would have been 80% wrong if written speculatively before G5 shipped.
