## Honest Feedback

1. **Diagnostic sequence could be faster** — I spent time fixing the embedded Python before discovering the venv. If I had checked the launch script first, I would have gone straight to the right environment. Lesson: when debugging bundled applications, check how they're actually launched before assuming which interpreter they use.

2. **Windows/WSL context switching adds friction** — Running PowerShell commands from WSL works but requires careful quoting. The backtick-newline escaping failed, and I had to try multiple approaches. This is inherent to the environment but adds time.

3. **The fix was fragile** — Installing packages with `--target` works but isn't the cleanest solution. The venv's pip being corrupted suggests the AMD bundle may have other issues lurking. Worth noting for future reference.

## Lessons Learned
1. Bundled applications often have multiple Python environments — always check which one the launcher actually uses
2. Virtual environments can have corrupted pip — `--target` flag from another Python can bypass this
3. The `python312._pth` file controls import paths for embedded Python — adding `Lib` and `Lib\site-packages` enables it to find installed packages
