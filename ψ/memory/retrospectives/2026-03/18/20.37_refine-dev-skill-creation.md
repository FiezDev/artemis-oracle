## AI Diary

The skill-creator workflow is elegant — spawn test agents in parallel, draft assertions while they run, grade outputs, then iterate. The benchmark showed a clear 75% improvement gap between with-skill (87.5%) and without-skill (12.5%).

What surprised me was how much the description optimization mattered. ## Honest Feedback

1. **Tool dependency friction**: The description optimization loop requires the `anthropic` Python SDK which wasn't installed. I worked around it by building a manual test harness, but this added ~15 minutes. The skill-creator workflow should either bundle dependencies or provide a fallback.

2. **Interactive skill testing complexity**: Since `refine-dev` is an interactive skill (6-step interview), testing required simulated user responses. This made the test setup more complex than a simple input/output skill. Future interactive skills might benefit from a standardized "simulated conversation" test format.

3. **Trigger vs execution separation**: The description optimization only tests triggering — not skill quality. A more comprehensive eval would test both "does it trigger correctly?" AND "does the skill produce good output?". The current workflow concedes these as separate concerns.

## Lessons Learned

1. **Description = UX for skills**: A skill can be perfectly functional but never get used if its description doesn't trigger correctly. Spending 30% of skill development time on description optimization is justified.

2. **Parallel testing scales well**: Running 4+ test agents simultaneously with simulated inputs is efficient. The grading script approach (regex checks on output files) is fast and reusable.

3. **Positive framing matters**: The skill's "translate negative to positive" constraint (e.g., "Don't use SQL strings" → "Use parameterized queries exclusively") produces better prompts. This pattern should be applied to skill descriptions too.
