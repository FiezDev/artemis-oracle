4. Things I discovered that changed the plan

After v1 of the prompts, I researched git history, Oracle memory, and past Claude Code sessions. Three findings were big enough to change the plan. They're all baked into the updated prompts.

🎯 RIC-303: LIFF code already exists

Original prompt said "design a new HTTP layer for LIFF token verification." But src/infrastructure/line/client.ts already has it:

  • verifyAccessToken(token) at line 87
  • getProfile(token) at line 114

Prompt now tells the executor to use the existing client, not rewrite it. Saves significant time.

🔑 RIC-301: Prod is SSM, not SSH

Per .github/workflows/README-CICD.md: prod deploys via aws ssm send-command, and the prod EC2's .env file lives on the host (not in CI secrets).

This is why the 2026-05-06 cutover dropped env vars — CI never had them, so when the host .env reset, there was no automatic recovery path.

Prompt updated: executor uses aws ssm start-session, not plain SSH.

⚠️ RIC-305: alert_delivery_state has LoRa history

The table was born in RIC-222 S3-A as the LoRa ACK state machine, not as a generic escalation table.

The UNIQUE(alert_id) constraint is load-bearing — added in RIC-191 code review (HIGH 2, commit 4e4906e) specifically because at-least-once redelivery was creating duplicate rows.

Prompt now requires ON CONFLICT (alert_id) DO NOTHING on publish-side INSERT, and warns the executor that the cron pipeline and LoRa ACK path share the table's lifecycle.

🧩 RIC-305: recordAlertAccess exists, no callers

The PDPA writer function at src/domains/audit/delivery-log.ts:79 is implemented, tested-by-implication, fire-and-log safe — but nothing calls it.

Original prompt said "write the writer." Updated prompt: "writer exists, wire the callers in each alert-namespace GraphQL resolver."

Two memory anchors added to RIC-305 from past retrospectives
  • "Disconnected code is invisible code" (2026-04-07 RIC-159/160 retrospective). Past mistake: files written but never registered in src/consumers/mod.ts or app startup were dead weight. Applied here: verify the DLX consumer is wired in startup, not just written.
  • "At-least-once redelivery creates duplicates" (2026-04-27 RIC-191 code review HIGH 2). Applied here: every alert_delivery_state INSERT must use ON CONFLICT; the UNIQUE constraint is there for a reason.
Cross-repo gotcha flagged in RIC-303 (informational, not blocking)

Per a Zenith Oracle reference doc: the RiceGuard frontend (separate repo) hardcodes severity values INFO | WARNING | CRITICAL, but the backend has migrated to P-tier (P1_CRITICAL | P2_HIGH | P3_MEDIUM | P4_LOW) in RIC-202. The LIFF profile pages may render stale severity strings. RIC-303 prompt notes this in the design doc as a follow-up ticket for Wasun.

  • The repo has no mock.module usage — it prefers integration tests against the TimescaleDB harness at src/test/harness/db-setup.ts. Original prompts assumed unit-with-mocks; updated to use the harness.
